RoboTwin 快速上手与自定义指南

这份文档把 RoboTwin 2.0 仓库整理成一张可操作地图：你先知道每块代码负责什么， 再沿着数据采集、任务开发、policy 部署三条线逐步改到自己想要的程度。

1. 仓库一句话理解

RoboTwin 是一个双臂机器人操作 benchmark 和数据生成平台。它把 仿真任务、专家示教、数据采集、 语言指令生成、policy 训练/部署/评测 放在同一个仓库里。

最核心的链路是： envs 任务环境 -> play_once 专家轨迹 -> collect_data 保存 hdf5 -> policy baseline 训练 -> eval_policy 部署评测。

2. 顶层结构地图

RoboTwin/ ├── README.md 项目说明、安装、使用、baseline 链接 ├── collect_data.sh 数据采集 shell 入口 ├── envs/ 仿真环境和任务定义，最核心 │ ├── _base_task.py Base_Task，任务运行时内核 │ ├── robot/ 机器人加载、关节、夹爪、规划器 │ ├── camera/ 相机封装 │ ├── utils/ 创建物体、保存数据、动作封装等工具 │ └── move_can_pot.py 等 每个任务一个文件 ├── task_config/ 采集/评测配置 │ ├── demo_clean.yml │ ├── demo_randomized.yml │ ├── _camera_config.yml │ └── _embodiment_config.yml ├── script/ Python 主入口脚本 │ ├── collect_data.py 采 seed、重放、保存 hdf5 │ └── eval_policy.py policy 评测主循环 ├── policy/ DP、DP3、ACT、RDT、pi0、pi05、VLA 等 │ └── Your_Policy/ 接入你自己模型的模板 ├── description/ 任务语言指令模板和生成工具 ├── assets/ 机器人、物体、纹理等资源 └── data/ 默认采集数据输出目录
目录/文件 作用 你通常怎么改
envs/*.py 具体任务：加载物体、专家动作、成功判断。 自定义任务、调整随机范围、改成功标准。
envs/_base_task.py 建 SAPIEN 场景、加载机器人/相机、采观察、执行动作。 一般先不要大改；需要新增控制模式或数据字段时再改。
task_config/*.yml episode 数、随机化、相机、数据类型、embodiment。 最适合入门调整。
script/collect_data.py 采集主流程：找成功 seed，再重放保存数据。 想改采集流程、保存逻辑、失败重试策略时看它。
script/eval_policy.py 评测主流程：加载任务、加载 policy、循环执行动作。 改评测次数、视频保存、instruction 选择、结果记录。
policy/Your_Policy 接自己模型的模板。 实现 get_model、eval、reset_model。

3. 数据采集主流程

最常用命令：

cd /mnt/disk1/shared_data/lzy/RoboTwin
bash collect_data.sh move_can_pot demo_randomized 0

它实际做的事情：

  1. collect_data.sh 接收 task_name、task_config、gpu_id。
  2. 调用 script/.update_path.sh 更新资源路径。
  3. 设置 CUDA_VISIBLE_DEVICES。
  4. 运行 python script/collect_data.py task_name task_config。
  5. 清理 data/task/config/.cache。

script/collect_data.py 的两阶段逻辑

阶段 1 搜索成功 seed

当 use_seed: false 时，脚本会不断尝试不同随机种子。 每个 seed 都会创建场景，运行任务的 play_once() 专家轨迹， 然后用 check_success() 判断是否成功。

成功的 seed 会写入 seed.txt，专家关节路径会保存到 _traj_data/episode*.pkl。

阶段 2 重放并保存数据

脚本读取已成功的 seed 和轨迹，关闭重新规划，重放专家路径。 这时 save_data: true，每隔 save_freq 保存 observation。

输出包括 data/episode*.hdf5、 video/episode*.mp4、scene_info.json。

这个设计的重点是可复现：先找到成功 seed 和轨迹，再重放保存数据。之后同样配置下可以复用 seed。

4. 任务环境怎么写

每个任务继承 Base_Task。建议先读 envs/move_can_pot.py，它结构很清楚。 

from ._base_task import Base_Task
from .utils import *

class my_task(Base_Task):
    def setup_demo(self, is_test=False, **kwargs):
        super()._init_task_env_(**kwargs)

    def load_actors(self):
        # 1. 创建或随机放置物体
        # 2. 记录目标位姿
        # 3. 设置禁止区域，避免杂物覆盖关键区域
        pass

    def play_once(self):
        # 专家示教：抓取、移动、放置等
        # 返回 self.info，供语言模板生成用
        pass

    def check_success(self):
        # 根据物体位置、姿态、夹爪状态判断任务是否完成
        return True
方法 职责 典型修改
setup_demo() 调用 _init_task_env_ 完成场景、机器人、相机、物体初始化。 大多数任务保持模板即可。
load_actors() 加载任务物体、随机位置、目标位置、禁止区域。 新增物体、改随机范围、改目标点。
play_once() 专家策略，通常用基类动作 API 拼出一段任务轨迹。 改专家动作顺序、换抓取方向、换放置路径。
check_success() 判断任务是否成功。 改容差、加姿态约束、加双臂状态约束。

常用任务动作 API

  • self.move(self.grasp_actor(actor, arm_tag="left"))：规划抓取某个物体。
  • self.move(self.place_actor(actor, target_pose=..., arm_tag="right"))：把物体放到目标位姿。
  • self.move(self.move_by_displacement(arm_tag, x=..., y=..., z=...))：末端按位移移动。
  • self.move(self.move_to_pose(arm_tag, target_pose))：移动到指定末端位姿。
  • self.open_gripper(arm_tag) / self.close_gripper(arm_tag)：开合夹爪。
  • self.back_to_origin(arm_tag)：回到初始位姿。
新建任务时，文件名、类名、命令里的 task_name 要一致。 例如 envs/my_pick_task.py 里应定义 class my_pick_task(Base_Task)。

5. 50 个任务具体是什么

RoboTwin 2.0 官方文档列出了 50 个双臂任务；当前本地仓库的 envs/ 与 description/task_instruction/ 中也包含 50 个任务文件。 这些任务不是随意堆物体，而是在测试双臂操作里的几个核心能力： 抓取搬运、双物体/多物体排序、工具或按钮交互、 容器放置、铰链/旋转/开合、双臂协作交接。

官方任务索引见 50 Bimanual Tasks； 本地任务源码都在 envs/<task_name>.py，语言模板在 description/task_instruction/<task_name>.json。

按能力分组理解任务

任务类别 包含任务 它在考什么 读源码重点
单物体抓取、搬运、放置 move_can_pot、move_stapler_pad、 move_pillbottle_pad、move_playingcard_away、 place_shoe、place_fan、 place_mouse_pad、place_empty_cup 识别目标物，选择手臂，抓起后移动到目标区域；通常是最适合入门调试的任务。 load_actors() 里的目标位姿和 check_success() 的距离阈值。
放入容器或放到支撑物上 place_object_basket、place_can_basket、 place_cans_plasticbox、place_container_plate、 place_object_scale、place_object_stand、 place_phone_stand、place_bread_basket、 place_bread_skillet、put_bottles_dustbin、 put_object_cabinet 目标区域通常是篮子、锅、盘子、柜子、秤、支架等，要求放置精度和碰撞规避。 目标物/容器的 bbox、姿态约束、是否需要先打开或避开容器边缘。
双物体、多物体组合 pick_dual_bottles、pick_diverse_bottles、 place_dual_shoes、place_burger_fries、 blocks_ranking_rgb、blocks_ranking_size、 stack_blocks_two、stack_blocks_three、 stack_bowls_two、stack_bowls_three 多目标选择、顺序约束、左右手协同、堆叠稳定性；对 policy 的长时序能力更敏感。 play_once() 的阶段划分，以及 check_success() 如何判断顺序/高度/相对位置。
按钮、开关、工具交互 click_alarmclock、click_bell、 press_stapler、turn_switch、 beat_block_hammer、stamp_seal、 scan_object 末端位置和姿态要对准按钮/工具接触点，有些任务用 stage_success_tag 记录阶段成功。 接触点、下压/敲击/扫描的运动方向，以及成功标记的触发条件。
铰链、开合、旋转、姿态调整 open_laptop、open_microwave、 rotate_qrcode、adjust_bottle、 shake_bottle、shake_bottle_horizontally、 lift_pot、dump_bin_bigbin 除了位置，还要控制角度、关节开合幅度、倾倒方向或摇晃轨迹。 check_success(target=...) 里的角度阈值，和 play_once() 中是否有备选轨迹。
双臂交接、悬挂、拿取 handover_block、handover_mic、 hanging_mug、grab_roller 强调两只手的相对位姿、交接时序和夹爪开合；对 action chunk 的时序误差更敏感。 左右臂分别承担的角色、交接点位姿、释放时机。

6. 配置系统

入门阶段优先改配置，不急着改代码。常用配置文件是 task_config/demo_clean.yml 和 task_config/demo_randomized.yml。 

render_freq: 0
episode_num: 50
use_seed: false
save_freq: 15
embodiment: [aloha-agilex]
language_num: 100
domain_randomization:
  random_background: true
  cluttered_table: true
  clean_background_rate: 0.02
  random_head_camera_dis: 0
  random_table_height: 0.03
  random_light: true
  crazy_random_light_rate: 0.02
camera:
  head_camera_type: D435
  wrist_camera_type: D435
  collect_head_camera: true
  collect_wrist_camera: true
data_type:
  rgb: true
  depth: false
  pointcloud: false
  endpose: true
  qpos: true
字段 含义 建议
episode_num 采集 episode 数。 调试时设小，比如 1-3；正式采集再放大。
render_freq 是否打开 viewer 渲染。 服务器或批量采集通常为 0。
save_freq 每隔多少步存一帧 observation。 影响数据时间分辨率和文件大小。
embodiment 机器人本体选择。 单项表示双臂同一机器人；三项可表示左右不同机器人加间距。
domain_randomization 背景、光照、杂物、桌高、相机扰动。 先用 demo_clean 跑通，再用 randomized 做鲁棒性。
data_type 保存 RGB、depth、pointcloud、qpos、endpose、segmentation 等。 按 policy 需要开关；点云和分割会增加存储成本。

7. 机器人与控制

机器人封装在 envs/robot/robot.py。它读取 embodiment 的 config.yml，加载 URDF，找到左右臂关节、夹爪关节、末端 link、相机 link， 并连接路径规划器。

Embodiment

task_config/_embodiment_config.yml 把名字映射到资源目录： aloha-agilex、piper、franka-panda、 ARX-X5、ur5-wsg 等。

Planner

规划逻辑在 envs/robot/planner.py，基类任务 API 会调用 left_plan_path、right_plan_path 等接口。

评测时动作控制

Base_Task.take_action(action, action_type="qpos") 是 policy 部署时最重要的接口。

  • qpos：动作格式是 [left_arm_joints + left_gripper + right_arm_joints + right_gripper]。
  • ee：动作格式是 [left_ee_pose xyz+quat + left_gripper + right_ee_pose xyz+quat + right_gripper]。

8. 数据格式

Base_Task.get_obs() 负责组装 observation。最终保存到 hdf5 的内容由 data_type 控制。 

{
  "observation": {
    "head_camera": {"rgb": ...},
    "left_camera": {"rgb": ...},
    "right_camera": {"rgb": ...}
  },
  "joint_action": {
    "left_arm": ...,
    "left_gripper": ...,
    "right_arm": ...,
    "right_gripper": ...,
    "vector": ...
  },
  "endpose": {
    "left_endpose": ...,
    "left_gripper": ...,
    "right_endpose": ...,
    "right_gripper": ...
  },
  "pointcloud": ...
}

RoboTwin 的部署接口怎么工作

文件/函数 作用 适配时最常改的点
policy/<Policy>/deploy_policy.yml 保存 checkpoint 路径、模型参数、评测配置等。 把 ckpt_dir、checkpoint_num、模型名、数据模态参数写进去。
policy/<Policy>/eval.sh 把命令行参数传给 script/eval_policy.py --overrides。 新增你模型需要的参数，例如 --model_path、--config_name。
get_model(usr_args) 加载模型，返回一个可推理对象。 解析 usr_args，初始化 tokenizer/VLM/diffusion head/缓存窗口。
encode_obs(observation) 把 RoboTwin 原始 observation 变成模型输入。 RGB resize/归一化、相机名映射、点云采样、qpos 拼接、语言指令注入。
eval(TASK_ENV, model, observation) 一个 episode 中反复取动作并执行。 决定是否一次输出 action chunk，执行时使用 qpos、ee 或 delta_ee。
reset_model(model) 每个 episode 开始前清空状态。 清空观测缓存、时间步、temporal aggregation、RNN hidden state。

DP baseline 的部署逻辑

policy/DP/deploy_policy.py 是一个很好的参考： 它把 head_camera、left_camera、right_camera 的 RGB 转成 channel-first，并把 joint_action["vector"] 作为机器人状态，然后让模型输出 qpos action。

接自己模型时，不建议一开始改 eval_policy.py 主循环。 更稳的做法是只改 policy/Your_Policy/deploy_policy.py。

10. 不同 policy 怎么适配

适配不同 policy 的本质，是把同一份 RoboTwin 数据转成各算法喜欢的 观测模态、动作格式、时间窗口 和 训练数据目录结构。官方文档的 policy 页面都遵循同一节奏： install -> process data -> train -> eval；本地仓库也按这个结构组织在 policy/ 目录下。

先按输入模态选 policy

Policy 主要输入 动作输出 数据采集要求 适合场景
ACT 三路 RGB + qpos qpos action chunk data_type.rgb: true，qpos: true 入门 baseline，时序动作块清晰，部署逻辑简单。
DP 三路 RGB + joint_action.vector qpos action chunk RGB 与 qpos 即可。 图像扩散策略，适合抓放类、容器放置类任务。
DP3 点云 + 机器人状态 qpos action chunk 必须采集 data_type.pointcloud: true；需要时开 use_rgb 点云颜色。 3D 几何关系重要的任务，例如放入容器、堆叠、狭窄空间放置。
RDT RGB/状态/语言，按 RDT 数据处理脚本组织。 通常转成 RoboTwin qpos 执行。 先采 RoboTwin 数据，再运行 policy/RDT/process_data_rdt.sh。 需要语言条件和机器人扩散 transformer 时。
pi0 / pi05 视觉 + 语言 + 状态，依赖对应模型仓库/权重。 动作序列，部署时转 RoboTwin action。 保留语言 instruction、RGB 和 qpos；注意 checkpoint 与 embodiment 维度。 想评估 VLA/通用策略模型在 RoboTwin 上的泛化表现。
DexVLA / TinyVLA VLM 图像 token + 语言 + diffusion/action head。 动作 chunk。 用 policy/DexVLA/process_data.py 生成 policy/DexVLA/data/sim_* 训练数据。 需要视觉语言骨干理解任务指令，同时用动作专家输出连续控制。
openvla-oft / LLaVA-VLA VLA 模型所需的图像、文本、状态。 按各自部署脚本转成 qpos/ee。 重点检查相机 resize、文本模板、动作归一化是否和训练一致。 接外部 VLA 模型或开放权重微调结果。
GO1 LeRobot 风格数据集 + GO-1 server。 通过 server/client 评测。 先 robotwin2hdf5.sh，再 hdf52lerobot.sh。 把 RoboTwin 数据迁移到 LeRobot/GO-1 训练生态。

四条适配路线

A. 图像策略 ACT / DP

保留 head_camera、left_camera、right_camera 的 RGB，并把 joint_action.vector 或左右臂 qpos 拼成状态。 ACT 的本地实现会把图像 resize 到 640x480，DP 则转为 channel-first 后除以 255。 

bash policy/ACT/process_data.sh beat_block_hammer demo_clean 50
bash policy/ACT/train.sh beat_block_hammer demo_clean 50 0 0
bash policy/ACT/eval.sh beat_block_hammer demo_randomized demo_clean 50 0 0

B. 3D 策略 DP3

DP3 的 encode_obs() 直接读取 observation["pointcloud"]。 因此采集前必须在任务配置里打开 data_type.pointcloud， 否则处理数据时容易出现空点云或除零错误。 

# task_config 中需要：
data_type:
  rgb: true
  pointcloud: true
  qpos: true

bash policy/DP3/process_data.sh beat_block_hammer demo_randomized 50

C. VLA 策略 DexVLA / pi0 / pi05

VLA 类策略比 ACT/DP 多一个语言链路：评测时可以通过 TASK_ENV.get_instruction() 获取当前 episode 指令。 适配时要确认训练数据里的 instruction、相机顺序、动作归一化、checkpoint action dim 与当前 embodiment 一致。 

# DexVLA 数据处理示例
cd policy/DexVLA
python process_data.py beat_block_hammer demo_clean 50

D. 外部数据生态 GO1 / LeRobot

GO1 不是直接吃 RoboTwin 原始目录，而是先把 RoboTwin 轨迹转成 HDF5， 再转成 LeRobot dataset。它适合你想把 RoboTwin 当数据源，接到外部训练框架里。 

cd policy/GO1
bash robotwin2hdf5.sh beat_block_hammer demo_clean 50
bash hdf52lerobot.sh beat_block_hammer demo_clean 50

自己接一个新 policy 的实用模板

  1. 复制 policy/Your_Policy 为 policy/MyPolicy，把 eval.sh 里的 policy_name 改成新目录名。
  2. 在 deploy_policy.yml 里放模型路径、checkpoint 名、是否使用语言、图像尺寸、action horizon 等参数。
  3. 在 encode_obs() 里统一相机名和尺度：RoboTwin 默认给 head_camera、left_camera、right_camera。
  4. 在 get_model() 里加载模型，并把动作归一化统计量、tokenizer、缓存窗口一起挂到 model 上。
  5. 在 eval() 里把模型输出转成 RoboTwin 支持的动作格式，优先从 qpos 跑通，再考虑 ee 或 delta_ee。
  6. 在 reset_model() 里清空 episode 级缓存；action chunk 模型尤其要清理历史动作和时间步。
最容易出错的是三件事：相机顺序和训练时不同、动作维度和 embodiment 不一致、 训练数据使用 clean 但评测配置切到 randomized。调试时先用同一个 task_config/ckpt_setting 做闭环，再测试跨配置泛化。

11. 自定义修改路线

入门级 改配置

  • 改 episode_num 控制采集数量。
  • 改 domain_randomization 控制背景、光照、杂物。
  • 改 camera 和 data_type 控制保存模态。
  • 改 description/task_instruction/*.json 增加语言模板。

中级 改任务

  • 修改 load_actors() 的物体、位置、目标点。
  • 修改 play_once() 的专家动作。
  • 修改 check_success() 的成功条件。
  • 复制已有任务文件，新建自己的任务。

高级 改资源和机器人

  • 往 assets/ 加新物体。
  • 新增 embodiment，配置 URDF、关节、夹爪、planner。
  • 修改 planner 或控制接口。
  • 新增 observation 字段或保存格式。

算法级 接模型

  • 在 policy/Your_Policy 写模型加载。
  • 实现 observation 编码。
  • 把模型输出转成 qpos 或 ee action。
  • 复用 eval.sh 和 script/eval_policy.py。

12. 推荐学习路径

  1. 先读 README.md，确认安装和数据下载方式。
  2. 用 demo_clean 跑一个小任务，建议把 episode_num 暂时改成 1。
  3. 看输出目录，理解 seed.txt、_traj_data、data/*.hdf5、video/*.mp4。
  4. 精读 envs/move_can_pot.py，把三个钩子函数和实际输出对上。
  5. 读 envs/_base_task.py 中的 get_obs、move、grasp_actor、place_actor、take_action。
  6. 复制一个任务，改物体位置或成功阈值，跑通采集。
  7. 再看 policy/Your_Policy/deploy_policy.py，接一个最小模型或随机动作 policy 做评测闭环。
